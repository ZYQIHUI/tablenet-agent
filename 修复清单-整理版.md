# TableNet-mini 修复清单（表格版）

## 1. 定位与验收原则

| 项目 | 定义 |
|---|---|
| `mini` 的含义 | 仅表示生成数据量、训练数据量和实验样本量小于论文规模 |
| 系统目标 | 多智能体机制、数据协议、评价口径和实验方法原则上与 TableNet 论文对齐 |
| 不允许的解释 | 不得以 `mini` 为理由省略论文核心 Agent、候选评价、反馈修复、增强或记忆机制 |
| 实现原则 | LLM 负责语义生成、评价和受限规划；确定性工具负责结构构造、验证、预算和终止 |
| 运行方式 | 同一套 Agent 和工作流必须支持远程 API、纯规则流、本地模型及混合模式，不为不同后端复制业务逻辑 |
| 论文结论原则 | 只有完成对应实验并保留证据后，才能声称复现或验证了该机制 |

### 支持的运行模式

| 模式 | 语义任务 | 结构任务 | 适用场景 | 结果标记 |
|---|---|---|---|---|
| `rule` | 规则模板、语料库和确定性生成器 | 确定性 Schema/Validator/Builder | 离线测试、稳定批量生成、回归测试 | `source=rule` |
| `api` | OpenAI-compatible 等远程模型 API | 确定性 Schema/Validator/Builder | 在线论文实验和外部模型对照 | `source=api` |
| `local` | 本地 Qwen 等模型直接推理或本地模型服务 | 确定性 Schema/Validator/Builder | 无第三方 API、数据不出本地 | `source=local_model` |
| `hybrid` | 按 Agent、错误类型、成本和可用性在 API、本地模型、规则间路由 | 确定性 Schema/Validator/Builder | 正式生成和容错运行 | 记录每次实际来源 |

运行模式只决定能力实现的选择，不改变 Topic、Header、Body、Checker 等 Agent 的职责和数据协议。

## 2. P0：统一结果协议与运行状态

| 编号 | 修复项 | 当前问题 | 目标实现 | 验收标准 | 性质 |
|:--|:--|:--|---|:--|---|
| P0-01 | 定义 `AgentResult` | Agent 主要只返回最终对象，LLM 失败和规则回退难以追踪 | 至少包含 `value/status/source/confidence/errors/metadata` | 能区分 `success`、`fallback`、`failed` | 可靠性基础 |
| P0-02 | 定义 `GenerationState` | Core 中状态分散在局部变量 | 保存 request、plan、schema、style、候选、评分、重试、失败和最终选择 | 任一失败均可定位到阶段、输入和原因 | 可靠性基础 |
| P0-03 | 修正静默回退 | `semantic_mode=llm` 失败后可能产出规则结果但来源不透明 | 记录 `requested_mode/actual_source/fallback_reason/model_name/attempt_count` | LLM 失败不得被统计为 LLM 成功 | 论文可信度必需 |
| P0-04 | 区分四类动作 | 当前重试通常等价于整表重新生成 | 分开记录 retry、repair、fallback、rebuild | 报告可分别统计四类动作及成功率 | 可靠性基础 |
| P0-05 | 建立最小执行 trace | 缺少 Agent 级输入、输出和耗时证据 | 使用结构化 JSONL 记录 request、candidate、Agent、模型、错误、动作和耗时 | 可根据 `request_id` 还原一次生成过程 | 论文实验必需 |

## 3. P0：建立 FillingAgent 反馈闭环

| 编号 | 修复项 | 当前问题 | 目标实现 | 验收标准 | 论文对应 |
|---|---|---|---|---|---|
| P0-06 | 新增独立 `FillingAgent` | Header、Body、Checker、选择和重试由 Core 直接串联 | FillingAgent 负责“生成候选 → 验证 → 评分 → 定位 → 修复 → 选择” | Core 不再包含具体填充重试细节 | Filling Agent |
| P0-07 | Checker 输出错误码 | 当前主要是自然语言 errors/warnings | 输出结构化错误码、位置、严重度和证据 | 每个拒绝结果至少有一个可执行原因 | Filling Checker |
| P0-08 | 错误到动作映射 | 检测结果不能稳定决定下一步 | 建立确定性 `error_type -> repair_action` 路由 | 相同错误在固定配置下进入相同修复路径 | Reflection |
| P0-09 | 消费 cell/column 分数 | 已生成的 `cell_scores/column_scores` 未用于修复 | FillingAgent 根据低分位置生成修复目标 | 单 cell/单列错误不会触发整表重填 | Selective regeneration |
| P0-10 | Header 局部重填 | HeaderAgent 只能整体填充 | 支持 `target_cells/target_columns/preserve_existing` | 非目标表头内容保持不变 | Selective regeneration |
| P0-11 | Body 局部重填 | BodyAgent 只能整体填充 | 支持 cell、column、row 范围修复并携带错误反馈 | 非目标正文 cell 保持不变 | Selective regeneration |
| P0-12 | 反馈式 repair prompt | 失败后缺少具体错误上下文 | 向生成 Agent 提供原结果、错误位置、预期类型和保留范围 | repair 调用与普通 retry 的 prompt 可区分 | Reflection |
| P0-13 | 修复预算与终止 | 只有粗粒度循环次数 | 限制 schema retries、filling repairs、LLM calls、耗时，并检测无改善 | 预算耗尽后明确拒绝并报告原因 | 可控执行 |

### 错误路由表

| 错误码 | 判定来源 | 默认动作 | 修复范围 |
|---|---|---|---|
| `STRUCTURE_INVALID` | Validator | 调用 Schema/Fallback Constructor | schema |
| `TOPIC_MISMATCH` | Checker/LLM evaluator | 重生成 topic 或相关内容 | topic 或全表语义 |
| `HEADER_INCOMPLETE` | Checker | HeaderAgent repair | 指定 header cells |
| `HEADER_BODY_MISMATCH` | Checker/LLM evaluator | 优先修复低分列正文，必要时修复表头 | column |
| `INVALID_CELL_TYPE` | Checker | BodyAgent repair | cell 或 column |
| `EMPTY_CELL` | Checker | 对空 cell 重填 | cell |
| `DUPLICATE_CONTENT` | Checker/deduplicator | 重填重复区域 | cell、row 或 candidate |
| `SEMANTIC_CONFLICT` | LLM evaluator | 修复冲突行；全局冲突则重建 Body | row 或 body |
| `LLM_CALL_FAILED` | Adapter | 明确规则回退、备用模型或终止 | 当前 Agent |
| `UNSUPPORTED_REQUEST` | Planner | 显式失败或记录后选择替代方案 | request/plan |

## 4. P0：对齐论文 Filling Checker

| 编号 | 修复项 | 当前问题 | 目标实现 | 验收标准 | 性质 |
|---|---|---|---|---|---|
| P0-14 | 三类核心维度 | 当前使用 title/header/body/topic 加权口径 | 对外统一为 structure correctness、topic relevance、semantic consistency | 每个候选均有三个独立分数 | 论文必需 |
| P0-15 | 结构硬门控 | 结构问题不能与语义分混合处理 | Validator 不通过即拒绝或进入结构修复 | 非法 coverage/span 候选不得入选 | 论文必需 |
| P0-16 | 最低维度总分 | 加权平均可能掩盖严重短板 | 论文口径使用 `overall=min(structure, topic, semantic)`；排名可另设辅助分 | 任一维度低于阈值不得直接通过 | 论文必需 |
| P0-17 | LLM 语义评价器 | 规则难以判断跨行一致性、真实性和复杂语义 | LLM 只评 topic relevance、header-body alignment、跨行一致性和合理性 | 保存评价 prompt、模型和原始响应 | 论文必需 |
| P0-18 | 隔离生成与评价 | 同模型同 prompt 可能自我偏好 | 使用独立 system prompt、隐藏生成过程、随机候选顺序，并与规则交叉检查 | 评价输入中不含候选生成轨迹 | 实验有效性 |
| P0-19 | 人工相关性实验 | 当前 Checker 未证明可替代人工排序 | 固定样本、多人评分，计算 Spearman/Pearson/Kendall | 报告样本数、行业、重复次数和方差 | 论文必需 |
| P0-20 | 修复迭代实验 | 当前循环次数不等于真实 correction iterations | 分结构、主题、语义扰动统计从失败到通过的修复轮数 | 可复现论文式 Avg Iters 指标 | 论文必需 |

## 5. P1：候选池与选择机制

| 编号 | 修复项 | 目标实现 | 验收标准 | 论文对应 |
|---|---|---|---|---|
| P1-01 | 候选数据模型 | 保存 `candidate_id/parent_id/source/transformation/schema/content/validation/scores` | 每个候选均可独立追踪 | 多版本生成 |
| P1-02 | 5 个普通内容版本 | 对每个结构生成 5 个未做变换的内容候选 | 配置为论文模式时稳定得到 5 个普通候选 | five without transformation |
| P1-03 | 4 个增强版本 | 分别生成 copy、delete、swap、alter 版本 | 每类操作至少对应一个候选并记录父候选 | four with transformations |
| P1-04 | 候选去重 | 检查 topic、header、body、schema signature 和纯样式差异 | 语义或结构等价候选不重复计数 | 多样性控制 |
| P1-05 | 候选排名与选择 | 依次应用结构门控、最低质量阈值、语义排名、多样性和成本规则 | 最终 metadata 保存选择与淘汰原因 | Filling Checker rank |
| P1-06 | 生成/增强谱系 | 保存候选父子关系和 transformation 参数 | 可从最终候选追溯到原始结构和每次操作 | 可观测性 |

## 6. P1：实现论文 5+4 增强

| 编号 | 修复项 | 目标实现 | 验收标准 | 注意事项 |
|---|---|---|---|---|
| P1-07 | span-aware 区域模型 | 基于 Validator coverage matrix 计算合并区域、边界和最小完整操作区域 | 任一操作不会拆分 rowspan/colspan | 结构基础 |
| P1-08 | `copy` 工具 | 复制完整 row、column 或 block，并修正坐标与跨度 | 操作后 coverage 完整且内容关系合理 | 结构增强 |
| P1-09 | `delete` 工具 | 仅在结构仍合法时删除完整区域 | 不产生空洞、越界或残缺 span | 结构增强 |
| P1-10 | `swap` 工具 | 交换兼容的 row、column 或 block | 坐标、表头关系和 span 均合法 | 结构增强 |
| P1-11 | `alter` 工具 | 按论文定义调整行级背景等视觉属性 | 不改变结构签名，视觉属性确有变化 | 视觉增强，勿计作结构修复 |
| P1-12 | 增强后双重验证 | 每次执行 transformation 后运行结构验证和语义验证 | 任一验证失败的增强候选被拒绝并记录 | 论文必需 |
| P1-13 | Fallback Constructor | 输入原 schema、错误、目标规模和保留内容，输出合法 schema | Fallback 后 Validator 必须通过 | 论文工具模块 |
| P1-14 | Schema 级修改 | 所有增强和修复作用于 `TableSchema/Cell`，最后统一生成 HTML | 不直接拼接或修补 HTML 字符串 | 项目实现约束 |

## 7. P1：Agent 职责与 Core LLM

| 编号 | 修复项 | 目标实现 | 验收标准 | 论文对应 |
|---|---|---|---|---|
| P1-15 | 新增 Request Planner | 将自然语言请求解析为 domain、数量、语言、结构、样式和预算约束 | 解析结果可验证且不直接执行任意代码 | Core LLM planning |
| P1-16 | 收紧 TopicAgent | 只生成 domain-relevant topic、scenario 和 language 语义结果 | 不擅自改变行列、simple、colored、lined | Topic generation LLM |
| P1-17 | 强化 SchemaAgent | 根据请求和语义场景规划行列、表头层级、方向和 span | 不再只依赖随机固定模板 | Schema planning |
| P1-18 | 内容感知 StyleAgent | 根据密度、语言、文本长度、层级、场景和渲染尺寸选择样式 | 渲染后无明显溢出，样式决策可追踪 | CSS generator |
| P1-19 | 未知请求显式处理 | 未支持结构不得静默随机替换 | 返回 unsupported 或明确记录 fallback alternative | 可控性 |
| P1-20 | 状态驱动 Core | 确定性 orchestrator 根据状态、错误和预算执行动作 | 权限、预算和终止不交给 LLM | 工程安全边界 |
| P1-21 | 受限 Core LLM | Core LLM 解释请求并在白名单动作中建议下一步 | 其建议需经 schema 校验和 orchestrator 执行 | 论文必需 |
| P1-21A | 统一能力接口 | Agent 依赖 `TopicGenerator/HeaderGenerator/BodyGenerator/SemanticJudge` 等协议，不依赖具体 API 或 Qwen 类 | 切换后端无需修改 Agent 业务代码 | 多后端基础 |
| P1-21B | 后端注册与路由 | 通过 registry/router 选择 API、本地模型或规则实现 | 支持按 Agent 配置主后端、fallback 和超时 | 多后端基础 |
| P1-21C | 统一模型调用记录 | API 和本地推理返回相同的调用结果结构 | 可统一统计模型、耗时、token、错误与回退 | 可观测性 |

## 8. P1：两级记忆

| 编号 | 修复项 | 目标实现 | 验收标准 | 论文对应 |
|---|---|---|---|---|
| P1-22 | Outer Memory | 保存多轮需求、约束变化和已确认偏好 | 同一会话后续请求能继承已确认条件 | 论文 Outer Memory |
| P1-23 | Inner Memory | 至少持久化已生成 topic，扩展记录 schema signature、header pattern 和拒绝原因 | 进程重启后仍能避免明显重复 | 论文 Inner Memory |
| P1-24 | 记忆边界 | 分离会话记忆、数据集多样性记忆和失败经验 | 三类数据具有独立生命周期 | 工程增强 |
| P1-25 | 并行一致性 | 多 worker 使用锁或共享存储更新去重信息 | 并发生成不出现大量重复 topic | 扩大规模时需要 |

## 10. P2：规模化工程增强

| 编号 | 修复项 | 何时需要 | 最小实现 |
|---|---|---|---|
| P2-01 | 自适应候选预算 | 固定 5+4 成本过高或做非论文生产模式时 | 简单表少候选，复杂 span 表多候选；论文模式保持 5+4 |
| P2-02 | 分数无改善终止 | 多轮修复成本明显增加时 | 连续若干轮无改善即停止 |
| P2-03 | 模型路由 | 多模型可用且需要控制成本时 | 轻模型生成、强模型语义修复、规则工具处理结构 |
| P2-04 | 任务队列与 worker 心跳 | 10K 以上或长时间并行生成时 | 队列、限流、断点续跑、失败恢复和输出冲突防护 |
| P2-05 | Trace replay | 失败样本难以复现时 | 固定输入、seed、模型参数和中间状态进行重放 |
| P2-06 | 记忆淘汰与压缩 | 持久记忆显著增大时 | TTL、摘要、索引和容量上限 |

## 11. 测试验收矩阵

| 编号 | 测试场景 | 预期结果 |
|---|---|---|
| T-01 | 单个 cell 类型错误 | 只修复该 cell，其他内容不变 |
| T-02 | 某列多项类型错误 | 只修复对应列 |
| T-03 | span 覆盖错误 | 只进入 Schema/Fallback 路径 |
| T-04 | LLM 请求失败 | 结果明确标记 fallback/failed，不伪装为 LLM 成功 |
| T-05 | 任一核心评分低于阈值 | 候选不得被加权平均掩盖后通过 |
| T-06 | 论文候选模式 | 每个结构得到 5 个普通候选和 4 个增强候选 |
| T-07 | 重复候选 | 不进入最终有效候选池，且有淘汰原因 |
| T-08 | copy/delete/swap 操作含 span 区域 | 不拆分合并区域，coverage 仍完整 |
| T-09 | alter 操作 | 结构签名不变，目标视觉属性发生变化 |
| T-10 | 增强后验证失败 | 候选被拒绝，不进入最终选择 |
| T-11 | Fallback Constructor | 输出 schema 合法，并保留要求保留的内容 |
| T-12 | 修复达到预算 | 明确终止并报告最后错误、轮次和成本 |
| T-13 | 同一 request trace | 可定位每个 Agent 的输入、输出、错误和动作 |
| T-14 | 多 worker 并发 | 无文件名冲突，topic/schema 重复率在设定阈值内 |
| T-15 | Core LLM 非法动作建议 | orchestrator 拒绝执行且状态不被直接修改 |
| T-16 | Checker 人工相关性 | 输出三类相关系数、样本量、重复次数和方差 |

## 12. 推荐实施顺序

| 阶段 | 实施内容 | 阶段完成标志 |
|---|---|---|
| 1 | `AgentResult`、最小 `GenerationState`、request/candidate ID、来源与回退记录 | 一次生成全程可追踪，静默回退消失 |
| 2 | 独立 FillingAgent、错误码、错误路由、局部 Header/Body 修复 | 形成可执行反馈闭环 |
| 3 | Checker 三维口径、结构硬门控、最低维度评分、LLM 语义评价 | 评价逻辑与论文对齐 |
| 4 | 候选池、5 个普通版本、去重、排名和选择 | 普通候选机制可稳定运行 |
| 5 | span-aware copy/delete/swap/alter、增强后验证、Fallback Constructor | 完整 5+4 候选可生成并验证 |
| 6 | Request Planner、Agent 职责调整、受限 Core LLM | 核心编排机制与论文对齐 |
| 7 | Outer/Inner Memory 和持久化去重 | 两级记忆可实验验证 |
| 8 | TEDS、5K 正式评价、真实 Direct 基线、真实数据泛化和各机制消融 | 代码能力转化为可信论文证据 |

### 目录职责与依赖边界

| 目录 | 职责 | 依赖约束 |
|---|---|---|
| `orchestration` | 工作流、状态流转、动作路由、依赖装配 | 可依赖 Agent 和能力协议，不直接依赖具体模型 SDK |
| `domain` | 请求、计划、Schema、候选、结果、错误和状态 | 不依赖其他业务目录，不执行 I/O |
| `capabilities` | 定义 Agent 所需能力协议和统一请求/响应 | 不知道能力来自 API、本地模型还是规则 |
| `planners` | Core、Topic、Schema 和 Style 规划 | 只通过能力协议调用语义能力 |
| `filling` | Header/Body 候选生成、去重、修复和选择 | 不直接实例化 API Client 或加载本地模型 |
| `evaluation` | 确定性结构验证和混合语义评价 | Structure Validator 永远不调用 LLM |
| `backends/api` | 远程 API 适配 | 负责鉴权、超时、重试和响应标准化，不包含 Agent 逻辑 |
| `backends/local` | 本地 Qwen 直接推理或本地服务适配 | 负责模型生命周期和推理，不包含工作流逻辑 |
| `backends/rule` | 各语义能力的确定性规则实现 | 作为正式 rule 模式或模型失败后的显式 fallback |
| `transformations` | span-aware 的 5+4 增强工具 | 只操作强类型 Schema，不直接修改 HTML 字符串 |
| `construction` | Schema/HTML 构造和 fallback | 不负责主题或正文语义生成 |
| `memory` | Outer/Inner Memory | 通过接口提供读写，不决定工作流动作 |
| `observability` | trace、模型调用和指标记录 | 只观察，不改变工作流状态 |
| `config` | 运行模式、后端路由、预算和论文参数 | 不保存密钥；密钥来自环境变量 |

### 后端路由示例

```yaml
mode: hybrid

capabilities:
  request_planning:
    primary: local_qwen
    fallback: rule
  topic_generation:
    primary: api_qwen
    fallback: local_qwen
  header_generation:
    primary: local_qwen
    fallback: rule
  body_generation:
    primary: local_qwen
    fallback: rule
  semantic_judging:
    primary: api_qwen
    fallback: local_qwen

backends:
  api_qwen:
    type: openai_compatible
    base_url_env: QWEN_API_BASE_URL
    api_key_env: QWEN_API_KEY
    model: qwen-plus
  local_qwen:
    type: transformers_qwen
    model_path_env: LOCAL_QWEN_MODEL_PATH
```

规则流不是异常兜底的代名词：`rule` 模式下它是明确选择的主实现；在 `api/local/hybrid` 模式下发生回退时，必须在 `AgentResult` 中记录请求模式、实际后端和回退原因。
