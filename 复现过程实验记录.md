## 2. 复现目标与边界

### 2.1 原论文目标

TableNet 的核心目标是利用多个智能体和工具协作，大规模生成具有以下特征的表格数据：

- 结构正确且可解析；
- 语义自然且内部一致；
- 样式具有足够多样性；
- 同时提供图像、HTML/结构真值及单元格级标注；
- 能够用于表格结构识别、内容识别和多模态模型训练。

### 2.2 本项目的实际复现范围

本项目采用资源受限的 **TableNet-mini** 路线，不直接复制论文中的完整 445K 数据规模。当前重点是先建立一个可验证、可扩展的最小闭环：

```text
需求规划
  -> 表格主题与结构生成
  -> 语义内容填充
  -> HTML 构建
  -> 浏览器渲染
  -> 单元格坐标回收
  -> 自动检查与失败重试
  -> 训练数据导出
  -> 多模态模型基线与 QLoRA 微调
```

本复现遵循的核心原则是：

> 规则和工具负责保证结构下限，智能体或大模型负责提升语义自然度，配置和数据增强负责扩大分布多样性。

因此，本项目不是让 LLM 独立完成所有步骤。对于行列跨度、HTML 合法性、单元格数量、坐标有效性等容易机械验证的内容，优先使用确定性程序约束；对于主题、字段名称和业务数据等开放内容，才允许 LLM 参与，并保留规则回退路径。

### 2.3 当前不应宣称的内容

目前不能声称：

- 已完整复现论文的 445K 数据集；
- 已实现论文全部智能体、记忆和增强机制；
- 已证明 QLoRA 提升 OCR 准确率或 TEDS；
- 已完成具有统计意义的真实在线 LLM Direct 对照实验；
- 已证明 FillingChecker 分数与人工评价具有显著相关性；
- 已完成正式规模的训练与下游评测。

## 4. 系统架构与模块职责

### 4.1 总体流水线

当前数据生成流程为：

```text
CoreAgent
  -> TopicAgent
  -> SchemaAgent
  -> StyleAgent
  -> HeaderAgent
  -> BodyAgent
  -> ValidatorAgent
  -> FillingChecker
  -> HtmlBuilder
  -> RendererTool
```

各模块职责如下。

| 模块 | 主要职责 | 关键输出 |
|---|---|---|
| `CoreAgent` | 调度整个样本生成过程，收集失败原因并触发重试 | 完整样本上下文、运行状态 |
| `TopicAgent` | 选择行业主题和业务场景 | topic、scenario |
| `SchemaAgent` | 生成行列规模、字段结构、合并关系 | schema、span 信息 |
| `StyleAgent` | 选择字体、背景、边框、对齐、间距等 | structured style fields |
| `HeaderAgent` | 生成标题、列头、分组表头、行头 | header cells |
| `BodyAgent` | 生成数据行、汇总行和语义一致的内容 | body cells |
| `ValidatorAgent` | 检查结构合法性与基本一致性 | validation result、errors |
| `FillingChecker` | 检查内容填充质量并决定是否接受 | filling score、failure reason |
| `HtmlBuilder` | 将结构化表格对象确定性转换为 HTML | `gt.txt`/HTML |
| `RendererTool` | 浏览器渲染并回收单元格坐标 | PNG、bbox、`cells.jsonl` |

### 4.2 数据契约

一个成功样本至少应产生或能追溯到以下内容：

- 表格图片；
- HTML 或 HTML 片段真值；
- 表格级元数据；
- 单元格文本；
- 单元格角色，例如 `title`、`column_header`、`row_header`、`body`；
- 单元格行列位置；
- `rowspan` 和 `colspan`；
- 图像坐标框 bbox；
- 结构配置与视觉配置；
- 验证和重试记录。

渲染后的 bbox 必须满足：

```text
x1 < x2
y1 < y2
x1 >= 0, y1 >= 0
x2 <= image_width
y2 <= image_height
```

### 4.3 LLM 的使用边界

系统支持 OpenAI-compatible 接口，但 LLM 是可选依赖。`semantic_mode` 支持：

- `auto`：有可用 LLM 时调用 LLM，失败时回退到规则；
- `llm`：明确要求使用 LLM，适合在线实验；
- `rule`：完全使用本地规则，适合离线复现和稳定测试。

此设计保证即使没有 API key，也能够完成结构生成、渲染、标注和大部分测试。

## 5. 已实现能力

### 5.1 八类平衡配置

系统覆盖以下三个二元维度：

- 结构：简单 / 复杂；
- 背景：纯色 / 有色；
- 边框：有线 / 无线。

三者组合形成 8 类基础配置。平衡抽样用于避免生成数据过度集中在某一种样式或结构上。16 样本平衡测试中，每类配置恰好出现 2 次。

### 5.2 复杂结构类型

当前已支持的复杂结构包括：

1. grouped columns：多个子列共享上层列组；
2. left headers：表格左侧存在行头；
3. body rowspan：正文区域纵向合并；
4. mixed headers：不同层级或方向的表头混合；
5. multi-level column header：多层列表头；
6. two-axis header：同时存在列头和行头；
7. summary-row colspan：汇总行横向合并。

这些类型由结构化 schema 控制，而不是在生成 HTML 后随机修改字符串。

### 5.3 语义场景

当前实现了 6 个电信业务语义场景：

- `base_station_maintenance`：基站维护；
- `broadband_installation`：宽带安装；
- `customer_complaints`：客户投诉；
- `network_coverage`：网络覆盖；
- `package_revenue`：套餐收入；
- `traffic_monitoring`：流量监测。

场景定义会约束字段名称、数据类型和合理值域，避免只生成“列1、列2”或互不相关的随机内容。

### 5.4 结构化视觉字段

视觉样式不再只保存成一段不可分析的 CSS，而是记录为可统计字段。已覆盖：

- 5 种字体；
- 4 档字号；
- 3 档单元格 padding；
- 3 类对齐方式；
- `plain`、`soft_fill`、`zebra` 三类背景；
- `thin`、`regular`、`strong` 三档边框粗细。

结构化记录使后续可以检查分布、定向补样，并开展样式消融实验。

### 5.5 失败重试与报告

批量生成不再因单个样本失败而整体终止。系统会：

1. 生成候选样本；
2. 执行 schema/结构检查；
3. 执行 FillingChecker；
4. 不通过时记录失败类型；
5. 在尝试上限内重新生成；
6. 最终报告目标数、尝试数、成功数和失败原因。

局部重试已经用于 schema 和 filling 阶段。这样既保留失败证据，也避免静默丢弃低质量样本。

### 5.6 单元格级标注

系统已能导出 `cells.jsonl`。每条记录可以包含样本标识、单元格文本、角色、行列索引、跨度和 bbox。当前 smoke test 中没有发现越界或退化 bbox。

尚未完成的是与论文完全一致的 S/C/H/V 独立标注套件，因此当前状态是“单元格级核心标注已实现，论文完整标注协议部分完成”。

## 6. 实验方法与评价定义

### 6.1 Smoke test 的定位

本文档中的 smoke test 用于确认：

- 程序能够完整运行；
- 输出文件存在且可解析；
- 结构和坐标满足基本约束；
- 配置分布符合预期；
- 训练不会立即 OOM、NaN 或崩溃。

Smoke test 不等同于正式性能实验。较小样本量只能发现明显错误，不能证明泛化能力。

### 6.2 结构保真度指标

结构实验至少报告：

- `Exact`：预测结构签名是否与目标完全一致；
- `Valid`：输出是否形成合法结构；
- `Avg score`：结构匹配平均得分；
- `Avg errors`：每个样本的平均结构错误数。

结构签名关注行列组织、标签和跨度。由于不同 HTML 序列可能表达同一个视觉表格，后续正式评测还需要补充标准化 HTML 和 TEDS。

### 6.3 单元格文本准确率

当前单样本检查使用：

```text
cell_accuracy = 正确单元格数 / 参考单元格总数
```

它只能衡量逐单元格文字是否一致，不能替代完整的结构指标。若模型增加 `<thead>`、`<tbody>` 或将 `<td>` 改为 `<th>`，文本准确率可能不变，但结构签名可能发生变化。

### 6.4 训练稳定性

训练阶段重点检查：

- 是否发生 CUDA OOM；
- loss 是否为 NaN/Inf；
- token 长度是否超过截断阈值；
- 可训练参数比例是否符合 QLoRA 预期；
- vision tower 和 projector 是否按配置冻结；
- adapter 是否成功保存并能重新加载推理。

## 8. 结构保真度小实验

### 8.1 实验问题

该实验用于初步比较两种路线：

- `agent_tool`：结构由智能体规划并由确定性工具生成；
- `llm_direct`：模拟模型直接输出结构。

实验分为简单/复杂、有内容/无内容四类 case，每类 7 个样本。

### 8.2 重要限制

本实验中的 `llm_direct` 是 **确定性的离线模拟器**，不是通过在线大模型 API 得到的真实 LLM 输出。它可以用于验证评测代码和观察预设错误模式，但不能作为论文中“智能体优于真实大模型”的最终证据。

### 8.3 结果

| 方法 | Case | 样本数 | Exact | Valid | Avg score | Avg errors |
|---|---|---:|---:|---:|---:|---:|
| `agent_tool` | 四类 case | 每类 7 | 7 | 7 | 1.0000 | 0.0000 |
| `llm_direct` 离线模拟 | `simple_with_content` | 7 | 7 | 7 | 1.0000 | 0.0000 |
| `llm_direct` 离线模拟 | `simple_without_content` | 7 | 7 | 7 | 1.0000 | 0.0000 |
| `llm_direct` 离线模拟 | `complex_with_content` | 7 | 0 | 0 | 0.8552 | 3.1429 |
| `llm_direct` 离线模拟 | `complex_without_content` | 7 | 3 | 3 | 0.9367 | 1.4286 |

### 8.4 可支持的结论

- 当前 Agent+Tool 路线在这组确定性测试中保持了完整结构约束；
- 离线 Direct 模拟器的复杂表格错误明显多于简单表格；
- 结构评测代码能够区分完全正确和存在跨度/层级错误的输出。

### 8.5 不可支持的结论

- 不能据此声称真实 LLM 在复杂表格上一定失败；
- 不能声称 Agent+Tool 已显著优于某个具体在线模型；
- 不能将离线模拟器结果写成 Qwen、GPT 或其他模型的实测结果。

下一步应固定 prompt、temperature、模型版本和样本集，运行真实在线 Direct 基线，并保存每个原始响应。

## 9. PAI-DSW 环境与目录说明

### 9.1 SSH 连接

本机使用私钥连接 PAI-DSW：

```powershell
ssh -i C:\Users\zy186\.ssh\id_rsa -p 1024 root@39.96.3.209
```

说明：

- 用户名：`root`；
- 公网地址：`39.96.3.209`；
- SSH 端口：`1024`，不是默认 22；
- `id_rsa` 是本机私钥，不能上传到仓库或发送给他人；
- 对应公钥需要配置在平台实例的 SSH 授权位置；
- 若直接执行 `ssh root@39.96.3.209`，会走 22 端口且不会显式选择该私钥，因此可能要求密码或连接失败。

### 9.2 磁盘挂载关系

服务器曾同时显示 `/mnt/workspace` 与 `/mnt/systemDisk`。检查结果表明它们对应同一个 `/dev/vdb` 工作盘，而不是两份独立磁盘数据。因此：

- 在 `/mnt/workspace` 写入文件会消耗该工作盘空间；
- 不能把两个挂载路径中的相似内容简单理解为两份可独立删除的数据；
- 项目统一使用 `/mnt/workspace` 路径，避免混淆；
- 删除前必须使用 `df -h`、`findmnt` 或 `mount` 再确认真实挂载关系。

工作盘名义容量约 98 GB。完成模型下载、训练缓存和输出后，约使用 26 GB，剩余约 73 GB。显示值的轻微差异来自文件系统格式化开销、单位换算以及系统保留空间。

### 11.3 转换时发现并修复的问题

修复方式：导出器在检测到缺少 `<table>` 根标签时，将原始行片段包装为：

```html
<table><tr>...</tr></table>
```

并新增回归测试，确保：

- 已有完整 `<table>` 时不重复包装；
- 只有 `<tr>` 片段时正确包装；
- 输出消息格式正确；
- 数据划分和路径生成稳定。

导出器相关测试共 4 项，全部通过。

## 16. 关键失败、原因与修复记录

### 16.1 SSH 要求密码或失败

**现象**：执行默认 SSH 命令时要求密码。  
**原因**：未指定 1024 端口或未指定正确私钥，SSH 没有使用平台中登记的公钥对应凭据。  
**修复**：使用：

```powershell

```

### 16.2 工作盘看似重复

**现象**：`/mnt/workspace` 和 `/mnt/systemDisk` 中的目录让人误以为有重复文件。  
**原因**：两个路径指向同一工作盘设备。  
**处理原则**：统一使用 `/mnt/workspace`；不要仅凭目录名执行删除，先检查挂载点和设备。

### 16.3 SFT HTML 根标签缺失

**现象**：真实 `gt.txt` 只有 `<tr>` 片段。  
**原因**：生成数据的真值格式与导出器原假设不一致。  
**修复**：导出时规范化为完整 `<table>...</table>`，并增加回归测试。

### 16.4 LLaMA-Factory 报 `KeyError: 'from'`

**现象**：数据预检无法解析 ShareGPT 消息。  
**原因**：`dataset_info.json` 缺少消息字段 tags 映射。  
**修复**：增加 `role_tag/content_tag/user_tag/assistant_tag` 映射。

### 16.5 基线文本较准但结构签名失败

**现象**：单元格 29/30 正确，结构仍不匹配。  
**原因**：模型使用了语义合理但不符合目标序列规范的 `<thead>`、`<tbody>`、`<th>` 等标签。  
**处理**：保留原始输出作为基线；通过 QLoRA 学习目标输出规范；未来同时报告标准化结构指标，避免只比较原始字符串。

## 19. 局限性与风险

### 19.1 数据规模风险

300 个合成样本只适合验证训练闭环。模型可能快速记住输出模板，loss 很低并不代表对新结构、新字体或真实扫描表格有泛化能力。

### 19.2 评价指标风险

当前单元格文本准确率和结构签名都比较粗糙。后续需要：

- HTML 规范化；
- 批量 TEDS；
- 结构与内容分开计分；
- 按简单/复杂、rowspan/colspan、样式类型分组报告；
- 保存失败样本用于误差分析。

### 19.3 合成分布风险

数据由规则和有限语义模板生成，可能存在字段模式重复、数字分布简单、视觉噪声不足等问题。模型在合成 test split 上表现好，不代表真实业务表格同样有效。

### 19.4 单样本结论风险

当前基线与 adapter 的直接比较只有一个样本。这个结果适合说明“格式学习现象”，不适合报告正式平均准确率。

### 19.5 存储与实例风险

PAI 试用实例和工作盘可能受费用、到期或释放策略影响。基础模型约 4.2 GB，训练目录和缓存会继续增长。重要的小文件应保留在 Git 或本地；大模型和数据应记录可靠的重新下载/生成方法。

### 19.6 凭据风险

SSH 私钥、API key 和平台凭据不得写入 Markdown、Git、训练配置或结果 JSON。文档只能记录私钥路径和配置方法，不能记录私钥内容。

### 20.1 第一优先级：完成 30 样本批量评价

目标：把当前单样本结论升级为完整 test split 结果。

具体任务：

1. 对 30 个 test 样本运行原始模型；
2. 对相同 30 个样本运行 QLoRA adapter；
3. 保存每个样本的原始输出、清洗输出和错误信息；
4. 计算 HTML validity、结构 exact match、单元格准确率；
5. 实现并计算 TEDS；
6. 按简单/复杂和跨度类型分组；
7. 输出逐样本明细和汇总 JSON/CSV；
8. 人工检查差异最大的若干样本。

这是当前最重要的下一步，因为它直接验证 QLoRA 除了单例格式对齐外，是否在整个测试集上稳定有效。

### 20.2 第二优先级：扩大数据规模

建议采用逐级放大：

```text
300 smoke -> 1K -> 5K -> 10K
```

每级都应记录：

- 数据生成成功率；
- 结构和样式分布；
- 重复率；
- token 长度；
- 训练时长和峰值显存；
- train/eval loss；
- test 指标；
- 典型错误。

只有当前一级验证通过，才进入下一级，避免把数据或评价错误放大。

### 20.3 第三优先级：补齐论文机制

后续实现项目：

- S/C/H/V 完整标注；
- 论文中的 5+4 数据增强策略；
- 两级记忆；
- 真实在线 LLM Direct 基线；
- FillingChecker 与人工评分相关性；
- 主动学习闭环。

这些工作应分别设计消融实验，不能只根据“代码已加入”判断有效。

## 22. 当前最终结论

当前项目已经从“能够生成一些表格”推进到“能够生成、验证、标注、导出，并用于多模态模型 QLoRA 训练”的完整最小闭环。数据生成端的结构约束、语义场景、视觉配置、重试和单元格标注均已得到 smoke test 验证；训练端已经完成数据转换、LLaMA-Factory 注册、原始模型基线、QLoRA 预检、正式 smoke 训练和 adapter 回载推理。

现阶段最可靠的模型侧发现是：**QLoRA adapter 学会了目标 HTML 输出格式，但在唯一已对比样本上没有改善单元格文本准确率。** 因此下一步不应立刻继续堆更大训练，而应先完成 30 个 test 样本的批量、同条件评价，加入 TEDS 和分组误差分析，再决定扩大到 1K、5K 或 10K 数据规模。
